# Snow 与 VCPChat 兼容依据

本文档用于说明：当我们把 `snow-cli` 当作 `VCPToolBox` 的 `TUI / IDE 代理前端` 使用时，应该向 `vcpChat` 学什么，不该学什么，以及当前 `snow-cli` 对 `VCPToolBox` 后端的兼容现状。

## 一、定位结论

`Snow CLI` 的目标不是变成另一个 `vcpChat`。

更准确的定位是：

- `vcpChat`：富前端、带渲染层、带前端环境注入
- `snow-cli`：面向 IDE 代理场景的 `VCP TUI`

因此，Snow 的兼容目标应当是：

- 兼容 `VCPToolBox` 的静态插件与静态语法
- 兼容 `VCPToolBox` 的核心显示协议
- 在 Snow 默认请求上下文不足时，补必要的 request bridge
- 不在 Snow 前端复刻 `VCPToolBox` 的主动插件执行体系

## 二、从 VCPChat 得到的关键事实

对照 `VCPChat` 前端源码后，可以得出三个结论：

### 1. VCPChat 不是 VCP 语法解释器

`vcpChat` 前端并没有实现这些 DSL 的语义解释：

- `::TagMemo`
- `::AIMemo`
- `::TimeDecay`
- `::RoleValve`
- `::Base64Memo`
- `vcp_dynamic_fold`

这些能力依然由 `VCPToolBox` 后端负责。

### 2. VCPChat 的前端职责是“协议适配”

它主要做四类事情：

- `system prompt` 原样透传
- 少量前端环境注入
- 上下文清洗
- 特殊返回块显示

也就是说，`vcpChat` 学的是“如何配合后端”，不是“如何抢后端的活”。

### 3. Snow 只该迁移 TUI 需要的那一层

对 Snow 有价值的不是把 `vcpChat` 的 DOM 和富文本渲染搬过来，而是搬它的分层思想：

- 能透传的就透传
- 真缺上下文时才 bridge
- 显示兼容只做显示兼容
- sanitizer 和 render 分离

## 三、VCP 能力分层

为了避免继续把“静态插件”“主动插件”“显示协议”混在一起，建议统一按下面四层理解。

### 1. 静态插件与静态语法

这类能力本质上依赖：

- `role.md / system prompt` 原文保真
- 后端变量替换与 DSL 解析

典型内容：

- `[[...]]`
- `《《...》》`
- `<<...>>`
- `{{...}}`
- `{{VCP...}}`
- `{{Var*}}`
- `{{Tar*}}`
- `{{Sar*}}`
- `::Time`
- `::Group`
- `::TagMemo`
- `::AIMemo`
- `::TimeDecay`
- `::RoleValve`
- `::Base64Memo`

这类能力对 Snow 的要求是：

- 不提前展开
- 不改写
- 不降级为 `user` 消息
- 保持为真正的 `system` 消息发给后端

### 2. 主动插件

这类能力依赖后端插件执行链或富前端交互链，Snow 不应在前端侧重复实现。

典型内容：

- 需要前端主动注入的交互插件
- 需要富 UI 容器才能完整工作的插件面板
- `vcpChat` 专属的画布、音乐、论坛等交互式前端壳

这类能力对 Snow 的要求是：

- 不要求前端实现
- 只要不要破坏后端协议即可

### 3. 显示协议

这类能力不一定改变后端行为，但会显著影响 TUI 可读性。

典型内容：

- `[@tag]` / `[@!tag]`
- `[--- VCP元思考链 ---]`
- `<<<[TOOL_REQUEST]>>>`
- `VCP-ToolResult`
- `ROLE_DIVIDE_*`
- `「始」...「末」`

这类能力对 Snow 的要求是：

- 至少保真，不误删、不误清洗
- 在 TUI 中尽量做结构化显示，而不是让整段协议文本裸奔

### 4. Request Bridge

这类能力只在 Snow 自身请求模型与 `vcpChat` 不同，导致后端信息不足时才需要。

当前最典型的就是：

- `::Time`

原因是：

- `vcpChat` 本身会把完整会话带给后端
- `snow-cli` 默认以本轮消息为主，历史不足

所以 `::Time` 在 Snow 侧需要 bridge，但这不代表 Snow 要为其他 DSL 也复制一套前端解释器。

## 四、当前 Snow 对 VCPToolBox 的兼容审计

### A. 已具备

#### 1. 静态语法保真链路

当前 `snow-cli` 已具备把自定义 `role.md / system prompt` 作为真正 `system` 消息发出的基础能力。

这意味着以下内容在请求链路上具备“保真透传”的前提：

- `[[...]]`
- `《《...》》`
- `<<...>>`
- `{{...}}`
- `{{VCP...}}`
- `{{Var*}}`
- `{{Tar*}}`
- `{{Sar*}}`
- `::Group`
- `::TagMemo`
- `::AIMemo`
- `::TimeDecay`
- `::RoleValve`
- `::Base64Memo`

结论：

- 这类静态插件与静态语法，Snow 当前不需要前端实现语义
- 但作为 `VCP TUI`，它已经具备了最关键的“原样送达后端”能力

#### 2. `::Time` request bridge

当前已实现：

- 仅在存在 `::Time` 语法时触发
- 仅改写 outbound request
- 不污染持久化 session
- 仅在保守链路上默认启用

这部分是 Snow 作为 `VCP TUI` 的必要补位，而不是对 `VCPToolBox` 后端能力的重复实现。

#### 3. `VCP元思考链` 显示兼容

当前已实现：

- 交互 UI 兼容
- Headless 模式兼容
- 与 `showThinking` 语义对齐

#### 4. `ghost anchor` 保真

当前已具备：

- 协议层 regex
- 纯逻辑回归测试

这说明 Snow 至少不会在兼容层主动把 `[@tag]` / `[@!tag]` 吃掉。

### B. 应保真，但不应在 Snow 前端重复实现

以下内容仍属于 `VCPToolBox` 后端职责：

- `::Group`
- `::TagMemo`
- `::AIMemo`
- `::TimeDecay`
- `::RoleValve`
- `::Base64Memo`
- `vcp_dynamic_fold`
- `{{VCP...}}` 的真实展开
- `{{Var*}} / {{Tar*}} / {{Sar*}}` 的真实替换

Snow 当前只要继续保证：

- `system prompt` 保真
- 不在发送前乱改这些文本

就已经满足 `VCP TUI` 的正确边界。

### C. 作为 VCP TUI 仍建议补齐

这些内容不是“必须把后端能力前端化”，但会直接影响 Snow 作为 `VCP TUI` 的可用性和观感。

#### 1. `TOOL_REQUEST / TOOL_RESULT` 显示层兼容

当前 `snow-cli` 源码中还没有发现对以下块的专门显示兼容：

- `<<<[TOOL_REQUEST]>>> ... <<<[END_TOOL_REQUEST]>>>`
- `VCP-ToolResult`
- `DailyNote` 结果块
- `ROLE_DIVIDE_*`

这意味着：

- 协议文本大概率能保真显示
- 但在 TUI 里还不够友好

#### 2. `「始」「末」` 保护式渲染

`vcpChat` 在渲染层会保护 `「始」...「末」` 之间的文本，避免其中内容被当作 HTML 或富文本结构误处理。

当前 `snow-cli` 还没有对应的显示层保护逻辑。

这对纯终端场景未必总是立刻出错，但一旦后续增强 Markdown / 富文本兼容，很容易变成隐患。

#### 3. 前端环境注入类能力

`vcpChat` 中存在一些前端注入能力，例如：

- `{{VCPChatCanvas}}`
- `{{VCPMusicController}}`
- `{{VarDivRender}}`

对于 Snow 来说，这些能力要区别对待：

- `Canvas / MusicController` 这类富前端能力：不属于 Snow 必做项
- `VarDivRender` 这类纯显示规范注入：可评估是否需要 TUI 版本的等价能力

也就是说，Snow 不需要追 `vcpChat` 的全部前端注入，只需要补对 `TUI / IDE 代理` 真有价值的那部分。

## 五、面向 VCP TUI 的最终边界

如果把 Snow 作为 `VCPToolBox` 的 `TUI / IDE agent front-end`，那么最合理的兼容边界是：

### 必须做

- `system prompt` 保真透传
- 静态插件与静态语法不被破坏
- `::Time` 这类确实因上下文模型差异而缺失的信息桥接
- `VCP元思考链`、`ghost anchor` 等关键显示协议兼容

### 建议做

- `TOOL_REQUEST / TOOL_RESULT / ROLE_DIVIDE / DailyNote` 的 TUI 显示增强
- `「始」「末」` 保护式显示

### 明确不做

- 在 Snow 前端复制 `VCPToolBox` 的主动插件系统
- 在 Snow 前端实现 `::Group / ::TagMemo / ::AIMemo` 等 DSL 语义
- 为了“看起来更聪明”而改写原始 `system prompt`

## 六、当前建议的后续优先级

优先级建议如下：

1. 补齐 `TOOL_REQUEST / TOOL_RESULT / ROLE_DIVIDE / DailyNote` 的显示层兼容
2. 增加 `「始」「末」` 的渲染保护
3. 继续维持 `::Time` bridge 的最小职责，不扩展成通用 DSL 前端解释器
4. 对 `VarDivRender` 一类“纯显示规范注入”单独评估是否值得做 TUI 版本兼容

一句话总结：

`Snow CLI` 应该做的是 `VCPToolBox` 的高保真 TUI，而不是削弱成普通聊天壳，也不是膨胀成另一个 `vcpChat`。
