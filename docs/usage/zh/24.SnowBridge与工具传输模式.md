# SnowBridge 与工具传输模式

本文档只回答两个问题：

- `local / bridge / hybrid` 现在到底怎么工作
- `SnowBridge` 侧需要配什么，哪些东西不需要手工重复配置

## 一、三种工具传输模式的真实行为

### 1. `local`

语义：

- 只使用 Snow 本地工具与本地 MCP 工具
- 不连接 `SnowBridge`
- 工具面里不应该出现 bridge 工具

适用场景：

- 只想验证 Snow 自己的工具链
- 需要把 `VCP mode` 聊天链和工具链隔离排查

### 2. `bridge`

语义：

- 只使用 `SnowBridge` 导出的远端工具
- 不保留本地工具
- 如果 bridge manifest 拉取失败，当前模式直接失败

适用场景：

- 专门验证 `SnowBridge` 到 `VCPToolBox` 插件链
- 需要确认远端工具描述、执行、中断与回灌链路

### 3. `hybrid`

语义：

- 同时加载本地工具和 `SnowBridge` 工具
- 重名时由本地工具优先
- 如果 bridge manifest 拉取失败，会自动退化为 local-only

适用场景：

- 既要保留 Snow 自己的工具，又要补上 VCP 侧远端插件
- 想让“读文件走本地，搜索/插件走 bridge”这类混合场景成立

## 二、推荐理解方式

可以把当前链路理解成两条并行通道：

- Chat：`Snow -> VCPToolBox /v1/chat/completions`
- Tools：`Snow -> local MCP` 或 `Snow -> SnowBridge WebSocket`

这意味着：

- `toolTransport` 只决定工具面，不决定聊天协议
- `backendMode` 只决定后端模式，不负责工具来源裁决
- `requestMethod` 在 `VCP mode` 下不应该再被理解成“聊天会真的直走某家 provider 原生端点”

## 三、最小配置示例

### 1. 本地工具模式

```json
{
  "snowcfg": {
    "baseUrl": "http://127.0.0.1:6005/v1",
    "requestMethod": "chat",
    "backendMode": "vcp",
    "toolTransport": "local"
  }
}
```

### 2. 纯 bridge 模式

```json
{
  "snowcfg": {
    "baseUrl": "http://127.0.0.1:6005/v1",
    "requestMethod": "chat",
    "backendMode": "vcp",
    "toolTransport": "bridge",
    "bridgeVcpKey": "123456",
    "bridgeAccessToken": ""
  }
}
```

### 3. 混合模式

```json
{
  "snowcfg": {
    "baseUrl": "http://127.0.0.1:6005/v1",
    "requestMethod": "chat",
    "backendMode": "vcp",
    "toolTransport": "hybrid",
    "bridgeVcpKey": "123456",
    "bridgeAccessToken": "",
    "enableVcpTimeBridge": true
  }
}
```

## 四、SnowBridge 侧必须知道的事情

### 1. 自动请求头

当前 Snow 会自动发送下面这组来源头：

- `x-snow-client=snow-cli`
- `x-snow-protocol=function-calling`
- `x-snow-tool-mode=local|bridge|hybrid`

在 `SnowBridge` WebSocket 请求上，还会额外发送：

- `x-snow-channel=bridge-ws`

所以：

- 你不需要再去自定义请求头配置里手工重复填写这些字段
- `SnowBridge` 的 `Require_Snow_Request_Headers=true` 是插件端校验项，不是 Snow 端手工录入项

### 2. 插件端典型配置

`SnowBridge` 侧通常至少要确认：

- `Bridge_Enabled=true`
- `Require_Snow_Request_Headers=true`
- `Allowed_Snow_Tool_Modes=bridge,hybrid`
- `Bridge_Access_Token` 仅在你需要额外鉴权时启用

如果你在插件端把 `Allowed_Snow_Tool_Modes` 只写成 `bridge`，那 `hybrid` 会被拒绝；如果写成 `bridge,hybrid`，那 `local` 请求不会被桥接器接受，这是预期行为。

## 五、当前路由语义里的几个关键点

### 1. `hybrid` 的优先级

当前实现明确采用：

- 本地工具优先
- bridge 工具补位

所以如果本地和 bridge 都有“读文件”之类重名工具，最终进入模型工具面的会是本地版本。

### 2. manifest 失败语义

当前不是所有模式都“自动兜底”：

- `bridge`：manifest 失败直接报错
- `hybrid`：manifest 失败退回 local-only，并记录告警
- `local`：不触碰 bridge

### 3. 旧协议不会回灌到工具面

当前 bridge translator 会把 `SnowBridge` 导出的工具描述清洗成模型可见的 function-calling 工具，不应该再把：

- `TOOL_REQUEST`
- `tool_name:`
- `「始」「末」`
- 旧示例块

直接暴露给模型。

## 六、排查建议

如果你遇到“工具丢失”“跑错工具”“桥接工具没有出现”这类问题，先按这个顺序看：

1. 先确认 `backendMode` 和 `toolTransport` 是否真的符合预期
2. 再确认 `bridgeVcpKey` 是否存在，且 `baseUrl` 是否指向 VCPToolBox 的 `/v1`
3. 再确认 VCPToolBox 侧 `SnowBridge` 已启用，且没有把当前 `x-snow-tool-mode` 拒掉
4. 最后再跑 `toolplane` 或黑盒脚本，不要先盯历史 session

推荐命令：

```bash
npm run test:vcp:toolplane
npm run test:vcp:blackbox -- --config scripts/fixtures/vcp-blackbox.local-bridge.json --mode local --mode bridge --mode hybrid
```
