# VCP 五层边界与 Snow Core Seam 说明

这份说明只回答一件事：Snow 在 VCP TUI 改造里，哪些地方能改，哪些地方不能改。

## 目标

Snow CLI 作为 VCP 的 TUI 时，需要兼容：

- VCP 请求边界
- VCP 工具注册边界
- VCP 返回显示边界

但这些兼容都不应该重新灌回 Snow Core，也不应该把 VCPToolBox 主链改成只为 Snow 服务。

## 五层职责

### 模型层

- 只看到统一的 function-calling 工具描述
- 不看到 `TOOL_REQUEST`、`tool_name:`、`<<<[TOOL_REQUEST]>>>` 这类旧协议示例
- 不看到重复工具定义

### Snow Core 层

- 负责对话流程、消息拼接、工具搜索、会话同步
- 只允许 seam 级接入兼容层结果
- 不直接处理 bridge manifest
- 不直接做工具路由决策
- 不直接感知 VCPToolBox 细节

### Translator 层

- 负责 `x-snow-*` 请求头
- 负责 `::Time` outbound bridge
- 负责 bridge manifest 到 model tools 的清洗与翻译
- 负责显示层兼容所需的文本转换

### Router 层

- 负责 `local / bridge / hybrid`
- 负责重名裁决
- 负责输出执行 binding
- 不负责协议清洗

### Executor 层

- Snow 本地工具负责本地 MCP 执行
- SnowBridge 负责桥接到 VCPToolBox 插件
- VCPToolBox 负责后端插件执行
- 不负责去重、翻译、模型面清洗

## 当前 seam 约束

当前 Snow Core 的 seam 落点是：

- [conversationSetup.ts](/H:/github/VCP/snow-cli/source/hooks/conversation/core/conversationSetup.ts)

它只允许通过：

- [toolPlaneFacade.ts](/H:/github/VCP/snow-cli/source/utils/session/vcpCompatibility/toolPlaneFacade.ts)

拿到以下结果：

- `tools`
- `servicesInfo`
- `duplicateToolNames`
- `toolPlaneKey`

也就是说，`conversationSetup.ts` 不应该再直接 import：

- `bridgeClient`
- `toolSnapshot`
- `toolRouteArbiter`
- `toolExecutionBinding`

这些模块都属于 translator/router/executor 侧，不属于 Snow Core。

## 为什么这样做

- 这样跟上游 Snow CLI 更容易，核心文件只保留 seam 级变化
- 这样 VCP 兼容层的回滚成本更低
- 这样问题定位更清楚，知道是 translator、router 还是 executor 出错
- 这样不会把 VCPToolBox 主链和 Snow Core 一起绑死
