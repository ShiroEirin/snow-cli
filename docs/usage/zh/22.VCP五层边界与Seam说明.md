# VCP 五层边界与 Snow Core Seam 说明

本文档只回答架构边界，不负责讲配置步骤和测试命令。

如果你要知道当前 `0.8` 测试标准分支怎么配置，请看 [`23.VCPTUI测试标准分支说明.md`](./23.VCPTUI测试标准分支说明.md)；如果你要看 `local / bridge / hybrid` 的工具面行为，请看 [`24.SnowBridge与工具传输模式.md`](./24.SnowBridge与工具传输模式.md)。

## 一、这条文档的范围

当前 `snow-cli` 作为 `VCP TUI` 时，需要兼容三类边界：

- 聊天请求边界
- 工具注册与执行边界
- 显示与会话转录边界

但这些兼容都不应该重新灌回 Snow Core，也不应该把 VCPToolBox 主链改成只为 Snow 服务。

## 二、五层职责

### 1. 模型层

职责：

- 只看到统一的 function-calling 工具描述
- 不看到旧 `TOOL_REQUEST` 母语示例
- 不看到重复工具定义

### 2. Translator 层

职责：

- 负责 `x-snow-*` 请求头
- 负责 `::Time` outbound bridge
- 负责 `SnowBridge manifest -> model tools` 的清洗与翻译
- 负责显示兼容所需的文本转换

对应模块主要位于：

- `source/utils/session/vcpCompatibility/mode.ts`
- `source/utils/session/vcpCompatibility/timeContextBridge.ts`
- `source/utils/session/vcpCompatibility/bridgeManifestTranslator.ts`
- `source/utils/session/vcpCompatibility/display.ts`

### 3. Router 层

职责：

- 负责 `local / bridge / hybrid` 的工具面裁决
- 负责重名去重与优先级
- 负责输出执行 binding

对应模块主要位于：

- `source/utils/session/vcpCompatibility/toolRouteArbiter.ts`
- `source/utils/session/vcpCompatibility/toolExecutionBinding.ts`
- `source/utils/session/vcpCompatibility/toolPlaneFacade.ts`

### 4. Executor 层

职责：

- Snow 本地工具负责本地 MCP 执行
- SnowBridge 负责桥接到 VCPToolBox 插件
- 执行器只消费 binding，不再自己判路由

### 5. Snow Core 层

职责：

- 对话流程
- 消息拼接
- 工具搜索
- 会话同步

边界要求：

- 只允许 seam 级接入兼容层结果
- 不直接处理 bridge manifest
- 不直接判定工具路由
- 不直接感知 VCPToolBox 插件细节

## 三、当前 seam 落点

当前核心 seam 仍然是：

- `source/hooks/conversation/core/conversationSetup.ts`

它应该只通过：

- `source/utils/session/vcpCompatibility/toolPlaneFacade.ts`

拿到工具面结果：

- `tools`
- `servicesInfo`
- `duplicateToolNames`
- `toolPlaneKey`

也就是说，`conversationSetup.ts` 不应该重新直接 import：

- `bridgeClient`
- `toolSnapshot`
- `toolRouteArbiter`
- `toolExecutionBinding`

这些模块属于 translator/router/executor 侧，不属于 Snow Core。

## 四、当前已经守住的边界

截至当前测试分支，已经收口的事实包括：

- `conversationSetup.ts` 通过 `prepareToolPlane()` 接 seam
- `toolExecutor.ts` 通过 execution binding 执行，不再在执行器里直接裁决本地还是 bridge
- `bridge manifest` 的清洗与参数翻译前移到 translator
- `local / bridge / hybrid` 的裁决集中到 router

## 五、当前仍存在的过渡态

当前还不能把边界说成“已经达到理想终态”，因为仍有少量 core 邻近链路引用兼容模块，例如：

- `contextCompressor.ts` 仍直接依赖 `display.ts` 与 `mode.ts`
- `streamFactory.ts`、`subAgentExecutor.ts`、`teamExecutor.ts` 等链路仍会感知 `resolveVcpModeRequest()`

这属于当前 `0.8` 分支的过渡态，而不是后续继续扩大改动面的理由。

换句话说：

- 可以承认当前实现还没达到最完美边界
- 但不能因此把更多 VCP 逻辑继续灌回 Snow Core

## 六、为什么要坚持这样分层

原因很直接：

- 更容易跟上游 Snow CLI 合并
- 兼容层回滚成本更低
- 出问题时更容易定位是 translator、router 还是 executor
- 不会把 VCPToolBox 主链和 Snow Core 继续绑死

## 七、当前防回退测试口径

这条边界目前主要靠以下测试与审计共同保护：

- `source/utils/session/vcpCompatibility/conversationSetupSeam.test.ts`
- `source/utils/session/vcpCompatibility/toolRouteArbiter.test.ts`
- `source/utils/session/vcpCompatibility/toolExecutionBinding.test.ts`
- 针对 `local / bridge / hybrid` 的黑盒脚本与会话抽样

其中最关键的一条依赖边界测试是：

- `conversationSetup.ts` 仍然通过 `toolPlaneFacade` 接入工具面
- `conversationSetup.ts` 没有重新直接依赖 `bridgeClient / toolSnapshot / toolRouteArbiter / toolExecutionBinding`

这类测试不是业务功能测试，而是边界防回退测试。
