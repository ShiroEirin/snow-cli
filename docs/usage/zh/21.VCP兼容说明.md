# Snow CLI VCP 兼容说明

欢迎使用 Snow CLI！本文档说明 Snow CLI 在接入 VCPToolBox / VCPChat 生态时的兼容边界、当前实现和后续约束。

## 目标

- 把 `Snow CLI` 作为 `VCPToolBox` 的 `TUI / IDE agent front-end` 来使用
- 让 `role.md` 中的 VCP 声明式语法可以原样交给后端解析
- 只在 Snow 必须补位的地方做兼容，不在前端复制整套 VCP 后端逻辑
- 保持请求体、持久化会话与显示层职责分离，避免再次出现“为了兼容而污染上下文”的问题

更具体的定位与和 `vcpChat` 的分工依据，见 [23.Snow与VCPChat兼容依据](./23.Snow与VCPChat兼容依据.md)。

## 兼容分层

### 1. System Prompt 保真

以下语法由 VCP 后端原生解析，Snow 不应提前展开、改写或降级为 `user` 消息：

- `[[...日记本...]]`
- `<<...日记本...>>`
- `《《...日记本...》》`
- `{{...日记本...}}`
- `[[VCP元思考...]]`
- `[[AIMemo=True]]`
- `{{VCP...}}`
- `{{Var*}}`
- `{{Tar*}}`
- `{{Sar*}}`
- `::Group`
- `::TagMemo`
- `::AIMemo`
- `::TimeDecay`
- `::RoleValve`
- `::Base64Memo`

当前 Snow 的做法是将自定义系统提示词作为真正的 `system` 消息发送，这一层以“原样透传”为最佳实践。

这意味着对于 `VCPToolBox` 的静态插件与静态语法，Snow 当前的正确职责是：

- 负责保真透传
- 不负责前端语义实现

### 2. Outbound Request 兼容

只有在后端确实需要上下文、但 Snow 默认发送内容不足时，才允许在发送前做桥接。

当前已实现：

- `::Time`
  - 通过 `source/utils/session/vcpCompatibility/timeContextBridge.ts`
  - 只改写本轮 outbound request
  - 不污染持久化 session
  - 默认仅对 `localhost` / `127.0.0.1` 一类保守链路启用

### 3. Response Display 兼容

Snow 需要兼容的不是“重写返回内容”，而是“不要误清洗，并在终端里正确显示”。

当前必须兼容：

- `[@tag]` / `[@!tag]`
  - 这是 VCP 的 ghost anchor，不能在 Snow 侧被误删
- `[--- VCP元思考链 ---] ... [--- 元思考链结束 ---]`
  - 这是 VCP 元思考链的明文输出格式
  - Snow 需要做视觉折叠/展开，而不是直接把整段当普通正文裸显

当前实现策略：

- 交互 UI:
  - 复用 `showThinking` 语义
  - `showThinking=false` 时折叠显示为单行摘要
  - `showThinking=true` 时渲染为结构化标题和正文
- 无头模式:
  - 默认折叠 `VCP元思考链`
  - 保留正文外的其他内容与 ghost anchor

当前实现曾经踩中过“把全文重写挂在实时渲染热路径上”的性能问题。后续修正方案与原因，见 [22.VCP元思考链显示改造单](./22.VCP元思考链显示改造单.md)。

当前已补齐：

- `<<<[TOOL_REQUEST]>>> ... <<<[END_TOOL_REQUEST]>>>`
- `VCP-ToolResult`
- `DailyNote` 结果块
- `ROLE_DIVIDE_*` 的非裸显处理
- `「始」「末」` 的保护式显示（在结构化块内转为字段显示）

### 4. Session Policy 兼容

VCP 后端会跳过以下伪 user 前缀：

- `[系统邀请指令:]`
- `[系统提示:]无内容`

Snow 后续如果继续扩展 VCP 兼容层，内部桥接消息也应遵守这套前缀语义，不能另造一套标记。

## 当前兼容状态

### 已完成

- `system prompt` 对静态插件/静态语法的保真透传前提
- `::Time` 请求前桥接
- `VCP元思考链` 显示层折叠
- `TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE` 的显示层兼容
- `ghost anchor` 保真回归测试
- `VCP` 内部邀请前缀的统一辅助函数
- `contextCompressor` 的 `VCP-aware` 转录与净化

### 建议后续补齐

- 仅在 TUI 确有价值时再评估 `VarDivRender` 一类纯显示规范注入

### 明确不在 Snow 侧重复实现

以下能力交由 VCP 后端处理，Snow 只负责保真透传：

- `{{agent:...}}`
- `{{toolbox:...}}`
- `{{Tar*}}`
- `{{Var*}}`
- `{{Sar*}}`
- `{{VCP...}}`
- `vcp_dynamic_fold`
- `::Group`
- `::TagMemo`
- `::TimeDecay`
- `::AIMemo`
- `::RoleValve`
- `::Base64Memo`

以下能力默认不作为 Snow 必做项：

- `vcpChat` 专属的前端环境注入（如 `{{VCPChatCanvas}}`、`{{VCPMusicController}}`）
- 依赖富前端容器的主动插件交互壳

## 测试建议

### 单元测试

优先覆盖以下纯逻辑：

- `::Time` bridge 触发/跳过条件
- `VCP元思考链` 折叠/显示转换
- `ghost anchor` 不被误删
- `VCP` 邀请前缀识别

### SSE HTTP 边界测试

Snow 可以通过 `SSE` 服务模式做黑盒测试：

- 启动方式：`snow --sse --sse-port <port>`
- 关键端点：
  - `GET /events`
  - `POST /session/create`
  - `POST /message`
  - `GET /health`

这类测试适合验证：

- `role.md` 中 VCP 声明是否被原样送出
- `::Time` 在真实请求链上是否触发 bridge
- `VCP元思考链` 是否在客户端最终显示中被折叠

但它不能替代单元测试，因为桥接和显示转换本身属于纯函数逻辑，回归风险更适合用快速测试覆盖。

## 后续扩展原则

- 新增 VCP 兼容时，优先落在 `source/utils/session/vcpCompatibility/`
- 不允许把兼容逻辑散落到各家 API provider 适配器里
- 不允许为了兼容而改写持久化历史
- 只在“Snow 必须补位”的场景下新增 bridge；能交给后端的，一律交给后端
