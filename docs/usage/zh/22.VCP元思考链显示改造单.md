# Snow CLI VCP 元思考链显示改造单

本文档用于记录 Snow CLI 在兼容 VCP `VCP元思考链` 显示时的修正方案、设计原因和实施边界，避免后续再次走回“为了兼容而把实时渲染链路做重”的老路。

## 一、问题定义

当前 Snow 已完成两类 VCP 兼容：

- 请求前兼容：`::Time`
- 显示层兼容：`VCP元思考链` 折叠/展开

其中，`::Time` 属于 outbound request bridge，职责清晰，风险可控。

本次暴露问题的是 `VCP元思考链` 的显示层兼容。现有实现虽然在功能上达成了“折叠显示”，但实现位置和时机不对，导致在 Snow 中出现明显的输出变慢、终端刷字卡顿、流式显示体感下降。

这不是 VCP 后端协议错误，也不是 `::Time` 桥接导致的问题，而是 Snow 侧显示兼容层放错了位置。

## 二、现状与根因

### 1. 当前 Snow 的实现方式

当前 Snow 在以下位置接入了 `VCP元思考链` 的显示转换：

- [MessageRenderer.tsx](/H:/github/VCP/snow-cli/source/ui/components/chat/MessageRenderer.tsx)
- [HeadlessModeScreen.tsx](/H:/github/VCP/snow-cli/source/ui/pages/HeadlessModeScreen.tsx)
- 旧的 `display.ts` 全文重写 helper（已移除；当前保留 parser / transcript helper）

核心做法是：

- 先拿到整段文本
- 再调用 `transformVcpContentForDisplay(...)`
- 用正则对全文做替换和重写
- 然后再交给 Markdown 渲染或控制台渲染

### 2. 这个实现为什么会慢

问题不在“正则存在”，而在“调用时机错误”。

当前实现把 `VCP元思考链` 的处理放进了通用显示函数里，这意味着：

- 只要 assistant 内容要显示，就可能触发一次全文扫描
- 文本越长，扫描成本越高
- 一旦 UI 在流式输出过程中频繁重渲染，就会反复对整段内容做转换
- `showThinking` 开关被复用进显示转换后，显示层逻辑和内容解析逻辑耦合得过深

结果就是：

- 请求本身未必变慢很多
- 但 Snow 的“显示体感”会明显变慢
- 用户会感觉“模型输出很慢”，实际上很可能是客户端渲染链路变重了

### 3. 为什么这不是 `::Time` 的锅

`::Time` 兼容目前在：

- [streamProcessor.ts](/H:/github/VCP/snow-cli/source/hooks/conversation/core/streamProcessor.ts)
- [applyOutboundMessageTransforms.ts](/H:/github/VCP/snow-cli/source/utils/session/vcpCompatibility/applyOutboundMessageTransforms.ts)
- [timeContextBridge.ts](/H:/github/VCP/snow-cli/source/utils/session/vcpCompatibility/timeContextBridge.ts)

其职责是：

- 仅在发送前检查是否需要 bridge
- 仅改写本轮 outbound messages
- 不污染 session 持久化
- 当前请求若不涉及 `::Time`，基本不会做额外改写

因此，当前“显示很慢”的主要怀疑对象不应是 `::Time` bridge，而应是 `VCP元思考链` 的显示兼容接法。

## 三、对照 VCPChat 的结论

本次修正必须以 VCPChat 为参考，而不是只按 Snow 当前实现继续修补。

### 1. VCPChat 的关键实现分层

参考：

- [messageRenderer.js](/H:/github/VCP/VCPChat/modules/messageRenderer.js#L43)
- [messageRenderer.js](/H:/github/VCP/VCPChat/modules/messageRenderer.js#L514)
- [streamManager.js](/H:/github/VCP/VCPChat/modules/renderer/streamManager.js#L177)
- [streamManager.js](/H:/github/VCP/VCPChat/modules/renderer/streamManager.js#L317)
- [streamManager.js](/H:/github/VCP/VCPChat/modules/renderer/streamManager.js#L856)
- [streamManager.js](/H:/github/VCP/VCPChat/modules/renderer/streamManager.js#L950)
- [contextSanitizer.js](/H:/github/VCP/VCPChat/modules/contextSanitizer.js#L185)
- [contextSanitizer.js](/H:/github/VCP/VCPChat/modules/contextSanitizer.js#L245)

VCPChat 的策略不是“每次显示都重写全文”，而是：

- streaming 阶段：只做轻量预处理
- final render 阶段：才做完整 prettify 和特殊块结构化处理
- sanitizer / export 阶段：单独做剥离或保留，不和实时显示混在一起

### 2. VCPChat 真正可迁移的不是代码，而是分层思想

应该迁移的是这三点：

- streaming 轻，final 重
- parser 与 renderer 分离
- 显示兼容与上下文净化分离

不应该迁移的是：

- 把 VCPChat 的 DOM 结构或浏览器式渲染逻辑硬搬到 Snow
- 在 Snow 的实时终端渲染链路里复刻 VCPChat 的重型 prettify

## 四、修正目标

本次改造的目标不是“继续让旧的 `display.ts` helper 更聪明”，而是把职责重新摆正：

- 保留 `::Time` 出站桥接
- 保留 ghost anchor 原样透传
- 保留 `VCP元思考链` 的折叠/展开能力
- 去掉实时渲染链路中的全文重写热路径
- 让 Snow 的显示兼容回到“轻量检测 + 最终结构化渲染”的模式

## 五、改造原则

### 1. 保真优先于美化

VCP 后端返回什么，Snow 默认应尽量保真显示。

只有在“会明显伤害可读性”的情况下，才做显示增强。

### 2. 流式阶段必须轻

流式阶段不允许引入全文正则重写、复杂 markdown 二次变换、结构化重排等重型逻辑。

### 3. 完整消息阶段才允许重处理

只有在消息已完整、内容不再增长时，才允许做：

- `VCP元思考链` 折叠
- 结构化标题/正文渲染
- 需要全文上下文的解析

### 4. 解析器与显示器分离

不能再把“是否存在 thought chain”“如何解析 thought chain”“如何显示折叠态”都塞进一个通用 helper 里。

### 5. 兼容层不得污染原始内容

任何 `VCP元思考链` 显示兼容都只能影响 display，不应改写：

- request body
- session 持久化内容
- 原始 `message.content`

## 六、具体改造方案

### 改造项 A：新增独立的 thought chain parser

建议新增文件：

- `source/utils/session/vcpCompatibility/thoughtChain.ts`

职责只做纯解析，不做 UI 渲染。

建议导出：

- `containsVcpThoughtChain(text: string): boolean`
- `parseVcpThoughtChains(text: string)`
- `stripVcpThoughtChains(text: string)`

其中：

- `containsVcpThoughtChain` 必须是 fast path
- 先用廉价判断，例如 `includes('[--- VCP元思考链')`
- 没命中时直接返回，不跑完整 regex

这样做的原因：

- 先把“是否需要处理”这一步做轻
- 避免所有消息都进入完整正则替换
- 让后续 UI 和 headless 都复用同一套 parser，而不是各自写一版字符串重写

### 改造项 B：降级当前 display helper 的职责

旧的 `display.ts` helper 既做解析又做显示文本重写，职责过重。

修正方向：

- 不再把它作为所有 assistant 内容的通用前置转换器
- 如果保留此文件，改成只服务“最终显示阶段”
- 不允许它继续挂在 streaming 高频渲染路径上

这样做的原因：

- 现在的慢点不是功能本身，而是接线位置
- helper 再聪明，只要挂在热路径里，仍然会拖慢体感

### 改造项 C：重构交互 UI 的接线方式

修改目标文件：

- [MessageRenderer.tsx](/H:/github/VCP/snow-cli/source/ui/components/chat/MessageRenderer.tsx)

目标策略：

- `streamingLine` 分支不做 `VCP元思考链` 结构化转换
- 普通 assistant / subagent 最终消息渲染时，先做一次 cheap detect
- 没命中 `VCP元思考链` 时，直接走原来的 markdown 渲染
- 只有命中时，才调用 parser
- 解析结果按“正文 + thought chain block”分别渲染
- `showThinking=false` 时，thought chain 渲染为折叠摘要
- `showThinking=true` 时，thought chain 渲染为结构化标题 + markdown 正文

这样做的原因：

- 把重处理从“每次显示”缩到“只有命中时才显示增强”
- 让普通消息继续走零额外损耗路径
- 行为更接近 VCPChat 的最终渲染模型

### 改造项 D：重构 HeadlessModeScreen 的接线方式

修改目标文件：

- [HeadlessModeScreen.tsx](/H:/github/VCP/snow-cli/source/ui/pages/HeadlessModeScreen.tsx)

目标策略：

- `renderConsoleMarkdown` 不再默认先对全文调用 `transformVcpContentForDisplay`
- 改为先做 cheap detect
- 只有命中 `VCP元思考链` 时才解析
- headless 默认折叠 chain，只输出单行摘要
- 正文部分继续正常进入 console markdown 解析

这样做的原因：

- 当前 headless 更容易在流式模式下频繁重复调用
- headless 最需要的其实不是“花哨显示”，而是“不要拖慢输出”
- 默认折叠是对的，但不应以全文重写作为代价

### 改造项 E：可选新增专用展示组件

建议新增：

- `source/ui/components/chat/VcpThoughtChainBlock.tsx`

职责：

- 只负责渲染 thought chain block
- 输入为结构化解析结果，而不是原始全文
- 管理 collapsed / expanded 语义

这样做的原因：

- 避免 `MessageRenderer.tsx` 再继续膨胀
- 把 VCP 特殊显示从通用消息渲染器中解耦
- 后续如果还要兼容别的 VCP block，也更容易继续扩展

### 改造项 F：导出/净化另走单独路径

如果后续 Snow 也需要做：

- 上下文裁剪
- 导出消息
- 存档摘要

则应参考 VCPChat 的 `contextSanitizer` 思路，单独处理，不允许复用实时显示逻辑。

原因：

- “显示给用户看”与“为了导出/上下文压缩而剥离内容”是两个不同职责
- 这两件事混在一起，最容易再次出现性能和行为回归

## 七、明确不做的事

本次修正中，以下内容不应顺手一起改：

- 不改 `::Time` bridge 主体逻辑
- 不改 outbound request 链路
- 不改 system prompt 保真策略
- 不新增任何 VCP 后端协议的前端模拟实现
- 不把 ghost anchor 做成可视化装饰
- 不把 `VCP元思考链` 重新写回 `message.content`

## 十、验收标准

满足以下条件才算本次修正完成：

- 普通消息的显示速度恢复到可接受水平
- `VCP元思考链` 在非 streaming 最终态仍能正确折叠/展开
- `::Time` 能力不回归
- `ghost anchor` 不丢失
- 原始消息内容不被污染
- 普通 assistant 消息不再为 thought chain 兼容付出额外热路径成本

## 十一、结论

本次问题不是“Snow 不该兼容 `VCP元思考链`”，而是“兼容方式不该放在实时全文重写热路径中”。

后续的正确方向不是继续修补旧的 `display.ts` helper，而是按 VCPChat 的分层重新落地：

- streaming 轻
- final 重
- parser 独立
- display 与 sanitizer 分离

这条原则比某一版具体代码更重要。后续只要继续遵守这条边界，Snow 再兼容更多 VCP 显示协议时，就不会再次把性能和职责一起拖进泥潭。
