# Snow CLI VCP 兼容回归测试清单

本文档用于记录 `snow-cli` 当前 VCP 兼容实现的回归测试要点，便于后续继续做 `VCP TUI` 迁移时快速复测。

前置约束：

- 本文档默认针对 `Snow VCP mode`，不用于替代 `Snow Native mode` 的 provider 原生回归
- 如果要判断某个问题是否属于“原生协议回归”还是“VCP 后端兼容回归”，请先对照 [`27.Snow Native与VCP Mode边界说明.md`](./27.Snow%20Native与VCP%20Mode边界说明.md)

## 一、测试目标

本轮需要重点守住三条边界：

- `::Time` 只做续问补位，不做前端时间语义扩写
- `VCP` 协议块只在最终态做结构化显示，不在流式阶段裸显协议壳
- 协议示例文本和压缩摘要不能因为兼容层而发生误判
- `VCP mode` 请求裁决不能静默剥掉 Snow 的工具能力
- `Anthropic / VCP mode` 不能再因为 tool schema 被 upstream `400`
- 无效工具纠偏不能进入无限重试
- 经过 `vcp-translation-server` 中转后，模型列表和中断链路仍然可用

## 二、建议测试矩阵

### 1. 显式时间词直通

目的：

- 确认 `今天 / 昨天 / 前几天 / 最近 / 近期 / recently / lately` 由 `VCPToolBox` 后端自己解析
- 确认 Snow 不再给这些宽时间词追加显式时间窗

建议输入：

- `查一下今天的日记`
- `查一下最近的日记`
- `show recently diary notes`

预期：

- Snow 发出的最后一条 user 不包含 `补充时间上下文：本轮 ::Time 检索按近7天时间窗理解...`
- `VCPToolBox-fork` 日志中的时间范围来自后端 `TimeExpressionParser`

### 2. 续问时间锚点补位

目的：

- 确认没有显式时间词的新一轮续问，仍能沿用上一轮时间锚点

建议对话：

1. `查一下昨天的日记`
2. `继续查一下`

预期：

- Snow 只在第二轮 user 末尾追加原始锚点桥接
- 追加文本类似：`补充时间上下文：本轮 ::Time 检索沿用上一轮用户提到的"昨天"。`

宽时间词续问也要覆盖：

1. `查一下最近的日记`
2. `继续查一下`

预期：

- 只沿用 `"最近"` 这个原始短语
- 不再展开成 `今天、昨天、前天...`

### 3. 语法讨论不误触发

目的：

- 确认用户在讨论 `ROLE.md` 或 `::Time` 语法本身时，不会被时间桥接污染

建议输入：

- `role.md 里的 [[Nova日记本::Time]] 是什么意思？`

预期：

- Snow 不追加 bridge 文本
- 后端不应出现与当前问题无关的时间召回

### 4. 协议示例代码块保护

目的：

- 确认 fenced code block 中的 `TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE` 示例不会被当作真实协议块解析

建议输入或测试文本：

````text
```text
<<<[TOOL_REQUEST]>>>
tool_name:「始」LightMemo「末」
<<<[END_TOOL_REQUEST]>>>
```
````

预期：

- UI 中应保留为普通代码示例
- 压缩摘要中也应保留示例文本，而不是被折叠成 `VCP-ToolRequest`

### 5. 流式协议壳抑制

目的：

- 确认 `TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE / VCP元思考链` 在流式阶段不会裸显协议边界

预期：

- 流式过程中不应看到 `<<<[TOOL_REQUEST]>>>`、`<<<DailyNoteStart>>>`、`ROLE_DIVIDE_*` 这类协议壳
- 最终消息完成后，应恢复为结构化显示

### 6. 压缩摘要兼容

目的：

- 确认 `contextCompressor` 会保留有用摘要，同时去掉协议噪音

预期：

- `ROLE_DIVIDE_*` 不进入摘要
- `VCP元思考链` 进入摘要时只保留折叠标签
- `VCP-ToolResult / VCP-DailyNote` 进入摘要时只保留关键信息

### 7. 多模型工具链黑盒

目的：

- 确认 `K2 / Gemini / Opus` 在 `Snow + VCP` 链路下都仍然具备真实工具调用能力
- 确认无效工具纠偏在重复失败时会终止当前轮，而不是无限回环

建议输入：

- `先调用 todo-get 获取当前 TODO，然后使用 filesystem-read 同时读取 AGENTS.md 和 snow-cli/ROLE.md，最后用一句话总结。不要修改文件，不要使用 terminal-execute。`

预期：

- `Gemini / Opus` 至少能发出真实 `tool_calls`，而不是输出 JSON 代码块冒充工具调用
- 如果模型反复输出拼接后的假工具名，Snow 会在达到纠偏阈值后直接终止当前轮
- 不应再出现长时间无上限的 headless 超时死循环

### 8. Anthropic schema 兼容黑盒

目的：

- 确认 `Anthropic-like` 模型通过 `VCP mode` 请求裁决时，不再因为 tool schema 被 upstream `400`
- 确认 `todo-get` 这类零参数工具、以及带联合类型参数的工具可以共存

建议输入：

- `先调用 todo-get 获取当前 TODO，然后使用 filesystem-read 同时读取 AGENTS.md 和 snow-cli/ROLE.md，最后用一句话总结。不要修改文件，不要使用 terminal-execute。`

预期：

- 过去对应的 `custom.input_schema` 400 不再出现
- `todo-get` 与 `filesystem-read` 可以在同一轮真实出现在 `tool_calls`
- `Gemini / K2` 路径不受这次 schema 清洗影响

### 9. 翻译层最小闭环

目的：

- 确认 `snow-cli -> vcp-translation-server -> VCPToolBox` 链路下的最小三端点已经打通
- 确认通过翻译器接入时，`Snow` 仍然按 `VCP mode` 工作，而不是误切回 `native`

建议配置：

```yaml
backendMode: vcp
requestMethod: chat
baseUrl: http://127.0.0.1:6107/v1
```

建议测试：

- 打开模型面板，确认可以正常拿到 `/v1/models`
- 发起一次普通对话，确认 `/v1/chat/completions` 正常返回
- 在流式输出中按 `ESC` 或主动断开客户端，确认翻译器会向上游触发中断清理
- 如果翻译器不是本地地址，再补一轮 `enableVcpTimeBridge=true` 的 `::Time` 续问回归

预期：

- 模型列表可正常展示
- 普通聊天无回归
- 断开连接后，应优先观察 `VCPToolBox` 的请求清理是否及时
- 远端翻译器场景下，显式开启 `enableVcpTimeBridge=true` 后，`::Time` 续问补位恢复正常

已知边界：

- `translator` 侧的 `/v1/interrupt` 路由和 `requestId` 透传可以单测证明是通的
- 但 `VCPToolBox` 当前实现会较早清理 `activeRequests`
- 因此在“流式响应已经开始后”再手动调用 `/v1/interrupt`，后端可能返回 `request not found or already completed`
- 这个现象当前应归类为 `VCPToolBox` 的生命周期限制，而不是 `translator` 没有转发 `requestId`

## 三、推荐日志观察点

如需联动 `VCPToolBox-fork` 检查，建议重点看：

- `LogAfterPreprocessors-*.txt`
- `LogOutputAfterProcessing-*.txt`

重点观察字段：

- `query`
- `timeRanges`
- `useTime`
- `results`
- `error.message`

## 四、结论标准

本轮兼容修正成立，需要同时满足：

- Snow 不再主动扩写宽时间词时间窗
- 续问补位仍然可用
- 协议示例代码块不会被误判
- 流式阶段不再裸显协议壳
- 最终态与压缩摘要仍保留 VCP 关键结构信息
- `Anthropic / VCP mode` 下原始工具集合经过 Snow 清洗后可稳定通过 upstream 校验
