# VCP Gateway 改造单

## 目标

在 **不修改 Snow 原生四协议语义** 的前提下，为 `snow-cli` 增加一层仅在 `VCP` 场景生效的请求兼容网关。

本次改造解决的问题是：

- Snow 原本按 `chat / responses / gemini / anthropic` 四套原生请求方式直连上游。
- `VCPToolBox` 的稳定入口，本质上更接近统一的 `chat/completions` 风格包装层。
- 在 `VCP` 后端链路里，不同模型族对 `tools` 的容忍度不同。
  - `Gemini` 往往需要保留工具声明。
  - `Claude / Opus / Sonnet / Haiku` 一类模型，保留工具声明更容易触发 `400`。

所以，正确做法不是修改 Snow 原生协议实现，而是新增一个 **VCP 扩展裁决层**。

## 设计原则

### 1. 不改四协议本体

保留 Snow 现有：

- `requestMethod = chat`
- `requestMethod = responses`
- `requestMethod = gemini`
- `requestMethod = anthropic`

这些原生模式在 **非 VCP 场景** 下继续保持原样。

### 2. VCP 兼容只做“本轮有效请求裁决”

Gateway 只裁决：

- 本轮 `effective request method`
- 本轮是否发送 `tools`
- 本轮是否发送 `tool_choice`

它不改：

- 已保存的 profile 语义
- Snow 自己的四协议定义
- provider 原生 thinking 参数结构

### 3. 默认保守，显式可覆盖

配置项采用三态：

- `undefined`：自动，仅对 `localhost / 127.0.0.1 / [::1]` 启用
- `true`：强制启用
- `false`：强制关闭

这样既适合本地 `VCPToolBox` 链路，也避免误伤普通远端 OpenAI/Anthropic/Gemini 服务。

## 兼容依据

### VCPChat 侧

`VCPChat` 发给 VCP 的请求体，核心是统一消息体加模型配置，而不是按四协议分别构造 provider-native body。

重点观察：

- `modules/chatManager.js`
- `modules/ipc/chatHandlers.js`

可见其稳定路径更接近统一的 `chat/completions` 风格请求。

### VCPToolBox 侧

`VCPToolBox` 的主聊天入口会对消息做预处理，但核心转发仍是统一的 `chat/completions` 包装层。

重点观察：

- `chatCompletionHandler.js`

它不会可靠地替 Snow 修正“不适合当前模型族的 tools/tool_choice”。

因此，修正应该发生在 Snow 自己的 VCP 扩展层。

## 本期实现范围

### 已做

- 新增 `VCP Gateway resolver`
- 新增配置项 `enableVcpGateway`
- 主链路与旁路链路统一复用 resolver
- 模型列表获取在 gateway 生效时走统一 `chat` 入口

### 本期不做

- 不新增第五种 Snow 原生协议类型
- 不在 Gateway 第一阶段统一各家 provider 的 thinking 强度参数
- 不在 Snow 里实现 VCP 主动插件执行器

## 当前裁决矩阵

Gateway 生效时：

- 有 `Claude / Anthropic / Opus / Sonnet / Haiku` 特征的模型
  - `effective request method = chat`
  - `tools = stripped`
  - `tool_choice = stripped`
- 有 `Gemini / LearnLM` 特征的模型
  - `effective request method = chat`
  - `tools = keep`
  - `tool_choice = keep`
- 其他模型
  - `effective request method = chat`
  - `tools = keep`
  - `tool_choice = keep`

Gateway 不生效时：

- 完全保持 Snow 原始请求行为

## 接入范围

为了避免“主对话修好了，旁路小模型继续炸”，resolver 需要覆盖：

- 主聊天流
- summary / compact / codebase review
- hooks 小模型
- sub-agent 执行链
- sub-agent 上下文压缩
- `/new-prompt`
- `/skills`
- models 列表获取

## 验证标准

### 单测

- gateway off 时不改原请求
- localhost auto 模式会启用 gateway
- anthropic-like 模型会 strip tools
- gemini-like 模型会保留 tools
- gateway 生效时 models 获取按 `chat` 入口裁决

### 黑盒

在 `Vcp-nova` 配置下联到 `VCPToolBox`：

- `Opus / Claude` 请求不再因工具声明触发 `400`
- `Gemini` 链路仍能保留工具能力
- `DebugLog` 中请求路径与请求体符合预期
