# Snow Native 与 VCP Mode 边界说明

本文档用于明确 `snow-cli` 在原生直连模式与 `VCPToolBox` 后端模式下的职责边界，避免继续把 `VCP Gateway` 理解成“四协议万能透传器”。

## 一、结论

- `Snow Native mode` 保留 Snow 原生的多协议直连能力。
- `Snow VCP mode` 单独作为一种后端工作模式存在，统一走 `VCP chat/completions` 语义。
- `VCP mode` 的目标是保住 `Snow` 的工具调用、终端交互和 VCP 语法兼容，不是承诺 `Claude / Gemini / Responses` 原生协议级行为完全等价。

## 二、为什么要拆成两种模式

在未接入 `VCP` 时，`Snow` 原本按自己的 provider-native 请求方式直连模型；接入 `VCPToolBox` 后，请求会进入统一的消息预处理、变量替换、RAG 注入和工具续轮链路。

`VCPToolBox` 自己的稳定入口是统一的 `/v1/chat/completions` 包装，而不是“把四协议原生语义一层不动地透传到上游”。因此，`Snow` 如果继续对外承诺“走 VCP 以后仍然等价于原生 Anthropic / Gemini / Responses 协议”，后续维护成本会持续失控。

正确做法是把“直连模型”和“经由 VCP 后端”视为两种不同产品模式。

## 三、模式定义

### 1. Snow Native mode

适用场景：

- 直接对接 `OpenAI / Anthropic / Gemini / Responses API`
- 需要保留 provider 原生协议字段和原生心智
- 需要完整使用 provider 专属能力

约束：

- 不主动引入 `VCP` 的后端语义
- 不假设存在 `::Time`、日记本占位符或 `VCP` 变量系统

### 2. Snow VCP mode

适用场景：

- `Snow` 作为 `VCPToolBox / VCPChat` 生态的 `TUI / IDE agent front-end`
- `ROLE.md` 中存在 `《《...》》 / [[...]] / {{...}} / ::Time / ::Group / ::TagMemo` 等 VCP 语法
- 需要通过 `VCP` 后端获得 `RAG / 变量 / TimeExpressionParser / 静态插件` 能力

约束：

- 请求入口统一使用 `chat/completions`
- 工具调用仍由 `Snow` 主控，不能被网关静默降级或剥离
- provider 原生专属字段不再承诺端到端等价保真

## 四、能力矩阵

| 能力项 | Snow Native mode | Snow VCP mode |
| --- | --- | --- |
| 传输协议 | `chat / responses / anthropic / gemini` 原生直连 | 统一 `chat/completions` |
| Snow 本地工具调用 | 强保证 | 强保证 |
| 工具调用主控权 | `Snow` | 仍由 `Snow` 主控 |
| `ROLE.md` / system prompt | 原生保真 | 原生保真，并允许 VCP 语法透传 |
| `::Time / ::Group / ::TagMemo / 日记本占位符` | 不承诺 | 强保证，交给 `VCP` 后端解释 |
| `VCP` 协议块显示兼容 | 不需要 | 强保证 |
| provider 原生思考参数 | 强保证 | 最佳努力，不承诺原样透传 |
| 通用参数 `stream / tools / tool_choice / max tokens` | 强保证 | 强保证 |
| provider 专属私有字段 | 可直接暴露 | 仅能映射已验证子集 |
| 模型列表获取 | 按原生协议入口 | 走 `VCP` 统一模型入口 |

## 五、参数管理原则

### 1. 参数分层

`Snow` 后续要把请求参数分成三层：

- 通用参数层：`system / messages / stream / tools / tool_choice / temperature / max tokens`
- 抽象能力层：例如统一的 `reasoningProfile = off / low / medium / high`
- provider 原生层：Anthropic、Gemini、OpenAI 各自的私有字段

### 2. 两种模式下的暴露规则

- `Native mode`：三层都可以暴露
- `VCP mode`：只暴露通用参数层，以及已经被验证可映射的抽象能力层
- provider 原生层在 `VCP mode` 下应隐藏、禁用，或明确标注为“最佳努力，不保证生效”

### 3. 对 thinking / reasoning 的处理

`Claude`、`Gemini` 等模型在原生协议下各有自己的思考控制参数；进入 `VCP mode` 后，不应继续假装这些字段仍然按原生协议完整生效。

更稳的策略是：

- `Native mode` 保留 provider 原生字段
- `VCP mode` 只暴露统一抽象项，例如 `reasoningProfile`
- 只有在 `VCP` 后端明确支持并完成黑盒验证后，才把统一抽象项映射到真实请求字段

## 六、实现边界

### 1. 当前 `VCP Gateway` 的正确定位

`VCP Gateway` 不是第五种 Snow 原生协议，它只是 `Snow VCP mode` 的请求裁决层。

它的职责应该是：

- 判定当前 profile 是否进入 `VCP mode`
- 在 `VCP mode` 下把本轮有效请求收束到 `chat/completions`
- 保持 `tools / tool_choice` 不被静默剥离
- 对 `Anthropic-like` 链路做 tool schema 兼容清洗

它不应该继续承担：

- 四协议长期统一透传的承诺
- provider 原生 thinking 参数的完全保真
- 第二套独立的工具执行主循环

### 2. Snow 与 VCP 的职责分工

`Snow` 负责：

- 对话主循环
- 工具调用与工具确认
- 终端 UI / TUI 交互
- `ROLE.md` 原样发送
- 必要的最小兼容桥接，例如 `::Time`
- `VCP` 协议块显示兼容

`VCPToolBox` 负责：

- 统一消息预处理
- 静态变量与静态插件
- `::Time` 的真实时间语义解析
- `RAG / TimeExpressionParser / 元思考链 / 日记本注入`

## 七、后续落地建议

- 配置层把 `enableVcpGateway` 逐步升级成显式模式概念，而不是单纯布尔开关
- UI 层明确区分 `Native mode` 和 `VCP mode`，不要继续让用户误以为 “VCP gateway = 原生协议继续完全生效”
- 黑盒测试按模式分开：
  - `Native mode` 回归只验证原生 provider 行为
  - `VCP mode` 回归重点验证工具调用、`::Time`、显示层和 schema 兼容
- 文档层统一措辞：
  - `Native mode` = 原生协议能力
  - `VCP mode` = VCP 后端兼容能力
  - 不再使用“万能网关”叙事

## 八、给接手者的判断准则

后续再遇到兼容需求时，先问三个问题：

1. 这是 `Snow` 原生协议能力，还是 `VCP` 后端能力？
2. 这个需求是否会破坏 `Snow` 的工具主控权？
3. 这个能力在 `VCP mode` 下是强保证、最佳努力，还是明确不承诺？

如果三个问题都答不清，就不要继续往 `gateway` 里堆逻辑。
