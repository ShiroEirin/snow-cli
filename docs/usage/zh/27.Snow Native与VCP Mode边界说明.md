# Snow Native 与 VCP Mode 边界说明

本文档用于明确 `snow-cli` 在原生直连模式与 `VCPToolBox` 后端模式下的职责边界，避免继续把 `VCP mode` 请求裁决层理解成“四协议万能透传器”。

## 一、结论

- `Snow Native mode` 保留 Snow 原生的多协议直连能力。
- `Snow VCP mode` 单独作为一种后端工作模式存在，统一走 `VCP chat/completions` 语义。
- `Snow VCP mode` 的推荐结构是“双通道直连”：聊天走 `VCPToolBox`，工具走 `SnowBridge`。
- `VCP mode` 的目标是保住 `Snow` 的工具调用、终端交互和 VCP 语法兼容，不是承诺 `Claude / Gemini / Responses` 原生协议级行为完全等价。

## 二、为什么要拆成两种模式

在未接入 `VCP` 时，`Snow` 原本按自己的 provider-native 请求方式直连模型；接入 `VCPToolBox` 后，请求会进入统一的消息预处理、变量替换、RAG 注入和工具续轮链路。

`VCPToolBox` 自己的稳定入口是统一的 `/v1/chat/completions` 包装，而不是“把四协议原生语义一层不动地透传到上游”。因此，`Snow` 如果继续对外承诺“走 VCP 以后仍然等价于原生 Anthropic / Gemini / Responses 协议”，后续维护成本会持续失控。

正确做法是把“直连模型”和“经由 VCP 后端”视为两种不同产品模式。

当前更推荐的做法是：

- 聊天面直接走 `Snow -> VCPToolBox /v1/chat/completions`
- 工具面单独走 `Snow -> SnowBridge WebSocket`

这样既不需要把 `Snow` 改成“四协议万能透传器”，也不需要继续深改 `VCPToolBox` 核心 `chat / interrupt` 主链。

## 三、模式定义

先明确三个互相独立、但经常被混用的配置轴：

- `backendMode`
  - 决定当前 profile 连接的是哪一种“后端产品模式”
  - `native` = 直连 provider
  - `vcp` = 进入 `VCPToolBox` 兼容模式
- `requestMethod`
  - 决定 Snow 北向请求使用哪套请求方言
  - 在 `native` 下，它直接对应 `chat / responses / anthropic / gemini`
  - 在 `vcp` 下，它只用于“兼容裁决”和能力映射，最终请求入口会统一收束到 `chat/completions`
- `toolTransport`
  - 决定工具面怎么接
  - `local` = 只接 Snow 本地工具
  - `bridge` = 在保留 Snow 本地工具的同时，额外接入 `SnowBridge`

这三层不是一回事：

- `backendMode` 不是 `requestMethod`
- `requestMethod` 不是 `toolTransport`
- `toolTransport` 也不应该反向决定聊天协议

### 1. Snow Native mode

适用场景：

- 直接对接 `OpenAI / Anthropic / Gemini / Responses API`
- 需要保留 provider 原生协议字段和原生心智
- 需要完整使用 provider 专属能力

约束：

- 不主动引入 `VCP` 的后端语义
- 不假设存在 `::Time`、日记本占位符或 `VCP` 变量系统

### 2. Snow VCP mode

适用场景：

- `Snow` 作为 `VCPToolBox / VCPChat` 生态的 `TUI / IDE agent front-end`
- `ROLE.md` 中存在 `《《...》》 / [[...]] / {{...}} / ::Time / ::Group / ::TagMemo` 等 VCP 语法
- 需要通过 `VCP` 后端获得 `RAG / 变量 / TimeExpressionParser / 静态插件` 能力

约束：

- 请求入口统一使用 `chat/completions`
- 工具调用仍由 `Snow` 主控，不能被网关静默降级或剥离
- `toolTransport = local | bridge`
  - `local`: 只使用 Snow 本地工具
  - `bridge`: 在保留 Snow 本地工具的前提下，额外加载 `SnowBridge` 导出的远端工具
- provider 原生专属字段不再承诺端到端等价保真

## 四、能力矩阵

| 能力项 | Snow Native mode | Snow VCP mode |
| --- | --- | --- |
| 传输协议 | `chat / responses / anthropic / gemini` 原生直连 | Chat 统一 `chat/completions`，Tools 可选 `local / bridge` |
| Snow 本地工具调用 | 强保证 | 强保证 |
| 远端 VCP 工具 | 不需要 | 通过 `SnowBridge` 接入，仍由 `Snow` 主控 |
| 工具调用主控权 | `Snow` | 仍由 `Snow` 主控 |
| `ROLE.md` / system prompt | 原生保真 | 原生保真，并允许 VCP 语法透传 |
| `::Time / ::Group / ::TagMemo / 日记本占位符` | 不承诺 | 强保证，交给 `VCP` 后端解释 |
| `VCP` 协议块显示兼容 | 不需要 | 强保证 |
| provider 原生思考参数 | 强保证 | 最佳努力，不承诺原样透传 |
| 通用参数 `stream / tools / tool_choice / max tokens` | 强保证 | 强保证 |
| provider 专属私有字段 | 可直接暴露 | 仅能映射已验证子集 |
| 模型列表获取 | 按原生协议入口 | 走 `VCP` 统一模型入口 |

## 五、参数管理原则

### 1. 参数分层

`Snow` 后续要把请求参数分成三层：

- 后端模式层：`backendMode`
- 请求方言层：`requestMethod`
- 工具传输层：`toolTransport / bridge ws url / bridge token / tool filter / fallback`
- 通用请求层：`system / messages / stream / tools / tool_choice / temperature / max tokens`
- 抽象能力层：例如统一的 `reasoningProfile = off / low / medium / high`
- provider 原生层：Anthropic、Gemini、OpenAI 各自的私有字段

### 2. 两种模式下的暴露规则

- `Native mode`：三层都可以暴露
- `VCP mode`：只暴露通用参数层，以及已经被验证可映射的抽象能力层
- `VCP mode`：工具传输层只暴露 `local / bridge` 这类后端能力选择，不再伪装成 provider-native 工具协议
- provider 原生层在 `VCP mode` 下应隐藏、禁用，或明确标注为“最佳努力，不保证生效”

当前代码实现里的对应关系已经固定为：

- `backendMode`
  - 由 `source/utils/session/vcpCompatibility/mode.ts` 决定是否进入 `VCP mode`
- `requestMethod`
  - 由各 provider 入口决定 Native 时走哪条北向链路
  - 在 `VCP mode` 下只参与兼容裁决，最后统一落到 `chat/completions`
- `toolTransport`
  - 由 canonical registry/router 决定工具面是否只走本地工具，还是同时注入 `SnowBridge`
  - 不再参与“猜服务归属”或“猜聊天协议”

### 3. 对 thinking / reasoning 的处理

`Claude`、`Gemini` 等模型在原生协议下各有自己的思考控制参数；进入 `VCP mode` 后，不应继续假装这些字段仍然按原生协议完整生效。

更稳的策略是：

- `Native mode` 保留 provider 原生字段
- `VCP mode` 只暴露统一抽象项，例如 `reasoningProfile`
- 只有在 `VCP` 后端明确支持并完成黑盒验证后，才把统一抽象项映射到真实请求字段

## 六、实现边界

### 1. 当前 `VCP mode` 请求裁决层的正确定位

`VCP mode` 请求裁决层不是第五种 Snow 原生协议，它只是 `Snow VCP mode` 下的请求入口收束器。

它的职责应该是：

- 判定当前 profile 是否进入 `VCP mode`
- 在 `VCP mode` 下把本轮有效请求收束到 `chat/completions`
- 保持 `tools / tool_choice` 不被静默剥离
- 对 `Anthropic-like` 链路做 tool schema 兼容清洗
- 在 `toolTransport = bridge` 时，把 `SnowBridge` 导出的远端工具注入 Snow 工具主循环
- 工具执行阶段只认 canonical registry/router，不再靠 `toolName` 前缀猜本地 / bridge / mcp 归属
- 旧 session 在读取时会做 tool identity hydration，把 `toolId / publicName / rawName` 补齐后再进入 replay

它不应该继续承担：

- 四协议长期统一透传的承诺
- provider 原生 thinking 参数的完全保真
- 第二套独立的聊天执行主循环

### 2. Snow 与 VCP 的职责分工

`Snow` 负责：

- 对话主循环
- 工具调用与工具确认
- `SnowBridge` 远端工具发现、执行与本地忽略迟到结果
- 终端 UI / TUI 交互
- `ROLE.md` 原样发送
- 必要的最小兼容桥接，例如 `::Time`
- `VCP` 协议块显示兼容

`VCPToolBox` 负责：

- 统一消息预处理
- 静态变量与静态插件
- `::Time` 的真实时间语义解析
- `RAG / TimeExpressionParser / 元思考链 / 日记本注入`
- `SnowBridge` 插件侧的远端工具导出、能力声明和取消确认

## 七、后续落地建议

- 配置层已经升级为显式 `backendMode = native | vcp`，不再依赖运行时 localhost 自动判定
- UI 层明确区分 `Native mode` 和 `VCP mode`，不要继续让用户误以为 “进入 VCP mode 后 provider 原生协议仍然完全等价生效”
- 黑盒测试按模式分开：
  - `Native mode` 回归只验证原生 provider 行为
  - `VCP mode` 回归重点验证工具调用、`::Time`、显示层和 schema 兼容
- 文档层统一措辞：
  - `Native mode` = 原生协议能力
  - `VCP mode` = VCP 后端兼容能力
  - 不再使用“万能网关”叙事

## 八、给接手者的判断准则

后续再遇到兼容需求时，先问三个问题：

1. 这是 `Snow` 原生协议能力，还是 `VCP` 后端能力？
2. 这个需求是否会破坏 `Snow` 的工具主控权？
3. 这个能力在 `VCP mode` 下是强保证、最佳努力，还是明确不承诺？

如果三个问题都答不清，就不要继续往 `VCP mode` 请求裁决层里堆逻辑。
