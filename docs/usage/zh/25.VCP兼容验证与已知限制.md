# VCP 兼容验证与已知限制

本文档记录当前 `VCP TUI` 0.8 测试标准分支的验证口径，不把临时黑盒样本伪装成正式产品承诺。

## 一、当前建议怎么验

### 1. 先做定向验证

优先跑这些定向检查：

```bash
npm run build
npm run test:vcp:toolplane
npm run test:vcp:blackbox -- --config scripts/fixtures/vcp-blackbox.local-bridge.json --mode local --mode bridge --mode hybrid
```

这组命令分别覆盖：

- 构建是否仍可通过
- 工具平面在 `local / bridge / hybrid` 下的注册结果
- fresh 会话下的聊天链与工具链黑盒

### 2. 再看四个基线场景

当前至少要分别验证：

- `native + local`
- `vcp + local`
- `vcp + bridge`
- `vcp + hybrid`

原因很简单：

- `native + local` 是 Snow 原生基线
- `vcp + local` 能隔离“聊天链进入 VCP，但工具仍是 Snow 本地”的行为
- `vcp + bridge` 能验证 SnowBridge 纯桥接工具面
- `vcp + hybrid` 能验证本地优先、bridge 补位的最终组合

### 3. 历史 session 不能代替 fresh 黑盒

当前分支已经吃过“历史坏 session 污染判断”的亏，所以建议：

- 用 fresh session 做最终判断
- 历史 session 只用来追日志，不用来下结论
- 黑盒脚本优先读取显式 fixture，不依赖当前环境碰巧存在的活动配置

## 二、当前已经守住的验证点

截至这条测试标准分支，已经作为“应继续保持”的验证点包括：

- `VCP mode` 下聊天请求统一收束到 `chat/completions`
- `Anthropic-like` 链路的 tool schema 兼容清洗
- `local / bridge / hybrid` 三种工具面裁决
- `hybrid` 下的本地工具优先
- `SnowBridge` 来源头隔离
- assistant 最终展示与持久化前的协议壳净化
- `::Time` 只做最小 outbound bridge，不把宽时间词前端扩写成固定时间窗
- `VCP元思考链 / TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE` 的显示兼容

## 三、当前还不能宣称通过的东西

下面这些还不能写成“已经正式完成”：

- `1.0` 正式版发布口径
- 所有 `team / subAgent` live 抽样全部完成
- 长时压力黑盒已经充分覆盖
- 所有模型家族在 `VCP mode` 下都与原生协议等价
- 任意版本 VCPToolBox 都能无缝兼容当前 SnowBridge

## 四、已知限制

### 1. 这还是测试标准分支

当前文档对应的是验证分支代码事实，不是 `main` 分支已经稳定发布的最终说明。

### 2. `VCP mode` 不是 provider 原生协议透传

进入 `VCP mode` 以后：

- `requestMethod` 不再代表聊天一定会走那条 provider 原生端点
- provider 私有参数只能按“已验证子集”理解
- 文档不应该再承诺“四协议原样透传”

### 3. `bridge` 模式依赖 VCPToolBox 侧的新 SnowBridge

当前 `bridge / hybrid` 不是通用 WebSocket 魔法：

- 需要 VCPToolBox 侧已经装好 `SnowBridge`
- 需要插件能识别 `x-snow-*` 请求头
- 需要 `bridgeVcpKey`

### 4. 模型侧异常不能直接当代码缺陷

这条分支之前出现过模型侧的异常工具名与异常输出，所以当前验证建议：

- 以 `glm-5` 这类相对稳定的回归基线为主
- 不把单一模型偶发输出直接写成架构结论
- 真要下结论，必须结合 fresh 黑盒和日志

## 五、1.0 前仍建议补的测试

如果后续要把这组文档升级到 `1.0`，建议至少补齐：

1. `team / subAgent` 的 live 黑盒抽样
2. 更长时的 `bridge / hybrid` 稳定性回归
3. VCPToolBox 侧版本差异的交叉验证
4. 不同模型家族在 `VCP mode` 下的对照样本
5. 从“旧 session 污染”到“fresh session 正常”的长期重复验证
