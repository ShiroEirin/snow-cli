# VCP TUI 测试标准分支说明

本文档描述的是当前 `snow-cli` 验证分支上的 `VCP TUI` 实现口径，而不是 `main` 分支的正式稳定承诺。

当前定位：

- 这是 `0.8` 测试标准分支
- 目标是给后续黑盒、兼容性审计和 `1.0` 收口提供统一基线
- 还不能宣称“正式版 VCP TUI 已完成”

如果你只想知道架构边界，请先看 [`22.VCP五层边界与Seam说明.md`](./22.VCP五层边界与Seam说明.md)。如果你要实际配置和使用，请继续看本文与 [`24.SnowBridge与工具传输模式.md`](./24.SnowBridge与工具传输模式.md)。

## 一、当前分支解决了什么

当前分支已经把以下能力收口成可测状态：

- `backendMode = vcp` 与 `backendMode = native` 的语义拆分
- `toolTransport = local | bridge | hybrid` 的真实路由与去重
- `Snow -> VCPToolBox /v1/chat/completions` 的聊天链
- `Snow -> SnowBridge WebSocket` 的工具桥接链
- `::Time` 的最小 outbound bridge
- `VCP元思考链 / TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE` 的显示兼容
- `bridge` 与 `hybrid` 模式的 Snow-only 请求头隔离
- 新鲜会话下的 `native + local`、`vcp + local`、`vcp + bridge`、`vcp + hybrid` 黑盒基线

这里要特别注意两件事：

- 当前真正有专门兼容实现的，主要还是 VCP 协议壳本身，也就是 `VCP元思考链 / TOOL_REQUEST / TOOL_RESULT / DailyNote / ROLE_DIVIDE`
- `[@tag] / [@!tag] / ::Group / ::TagMemo / ::AIMemo / ::TimeDecay / ::RoleValve / ::Base64Memo / [[...]] / <<...>> / 《《...》》 / {{Var*}} / {{Tar*}} / {{Sar*}}` 这一批大多只是 pass-through 或上游被动处理结果，不能在这条测试标准分支文档里写成“Snow 已正式承诺完整兼容”

当前分支没有解决，也不应该在文档里伪装成已经解决的内容：

- `1.0` 正式版级别的全链路长期稳定性
- 所有 `team / subAgent` live 场景都已完成验收
- `VCP mode` 下对各家 provider 原生私有字段的完全等价保真
- 让 `Snow` 输出旧 `TOOL_REQUEST` 母语协议来代替 function-calling

## 二、你需要准备什么

### 1. Snow 侧最小前提

当前实现依赖这几个核心配置字段：

- `baseUrl`
- `requestMethod`
- `backendMode`
- `toolTransport`
- `bridgeVcpKey`（仅 `bridge / hybrid` 需要）
- `bridgeAccessToken`（可选）
- `enableVcpTimeBridge`（可选）

最小示例：

```json
{
  "snowcfg": {
    "baseUrl": "http://127.0.0.1:6005/v1",
    "apiKey": "your-key-if-required-by-upstream",
    "requestMethod": "chat",
    "backendMode": "vcp",
    "toolTransport": "hybrid",
    "bridgeVcpKey": "123456",
    "bridgeAccessToken": "",
    "enableVcpTimeBridge": true
  }
}
```

注意：

- `backendMode` 决定是否进入 `VCP mode`
- `requestMethod` 在 `VCP mode` 下主要用于兼容裁决，不代表聊天最终还会走 `responses / anthropic / gemini` 原生端点
- `toolTransport = bridge` 或 `hybrid` 时，`bridgeVcpKey` 是必填项

补充说明：

- `::Time` 当前只表示“最小 outbound 时间锚点 bridge”
- 它不会在 Snow 侧主动把宽时间词展开成完整时间窗，也不是通用时间语义改写器

### 2. VCPToolBox 侧最小前提

当前链路默认要求 VCPToolBox 侧已经安装并启用了新版 `SnowBridge` 插件，并且插件能消费 Snow 自动发送的请求元数据。

最小前提包括：

- VCPToolBox 提供 `/v1/chat/completions`
- SnowBridge 插件已启用
- SnowBridge 的 `Bridge_Enabled=true`
- 如启用 Snow-only 校验，则 `Require_Snow_Request_Headers=true`
- `bridge / hybrid` 模式允许的 `x-snow-tool-mode` 与 Snow 配置保持一致

## 三、配置轴的真实语义

### 1. `backendMode`

- `native`：保持 Snow 原生聊天协议语义
- `vcp`：进入 `VCP mode`，聊天链统一收束到 `chat/completions`

### 2. `requestMethod`

- `native` 下：真实决定走 `chat / responses / anthropic / gemini`
- `vcp` 下：只参与兼容裁决和部分 schema 兼容逻辑

### 3. `toolTransport`

- `local`：只保留 Snow 本地工具
- `bridge`：只保留 `SnowBridge` 暴露的远端工具
- `hybrid`：本地工具与 bridge 工具同时进入工具面，重名时 `local > bridge`

这一点和旧文档最大的差异在于：当前代码下 `bridge` 模式不是“本地工具加 bridge 工具”，而是纯 bridge 工具面；只有 `hybrid` 才是混合工具面。

## 四、当前代码已经自动做的事

下面这些东西已经由当前分支自动处理，不需要你再手工配置一遍：

- `VCP mode` 聊天请求自动附带：
  - `x-snow-client=snow-cli`
  - `x-snow-protocol=function-calling`
  - `x-snow-tool-mode=local|bridge|hybrid`
- `SnowBridge` WebSocket 请求自动附带：
  - 上述 3 个头
  - `x-snow-channel=bridge-ws`
- `bridge / hybrid` 模式下自动使用 `bridgeVcpKey` 拼接 WebSocket 地址
- `hybrid` 模式下自动做工具去重，且本地工具优先

所以如果你看到 `SnowBridge` 的 `Require_Snow_Request_Headers` 之类配置，不代表还要在 Snow 自定义请求头里再手填一次；这部分已经由代码自动注入。

## 五、当前已经通过的基线

截至当前验证分支，已经作为“通过样本”留痕的基线包括：

- `native + local`
- `vcp + local`
- `vcp + bridge`
- `vcp + hybrid`

其中最新一轮 fresh 黑盒已经再次确认：

- `local` 仍然命中 `filesystem-read -> source/cli.tsx`
- `bridge / hybrid` 都能命中 `vcp-servercodesearcher-searchcode(query=SnowBridge)`
- bridge 搜索类黑盒现在验证的是“工具返回 success，assistant 复述当前 workspace 的首条命中路径”，不再把历史上某个固定文件路径当成唯一正确样本

并且当前代码已经补齐了：

- `hybrid` 下的 `local > bridge` 裁决
- `hybrid` 下 bridge manifest 失败时回退为 local-only
- `bridge` 下 manifest 失败时直接报错，而不是静默退化
- assistant 正文保存前的协议壳净化
- 黑盒脚本的配置入口与会话文件读取竞态容错

## 六、为什么还是 0.8

当前不能把这套文档写成 `1.0`，原因很明确：

- `team / subAgent` live 抽样还没有完全收口
- 仍然需要更多 fresh 会话与长链路压力测试
- 当前实现虽然守住了五层主边界，但还存在少量 core 邻近链路的过渡性引用
- `VCP mode` 下的 provider 私有能力还没有做到“文档可承诺的全面兼容”

因此，这一组文档的正确读法是：

- 把它当成当前测试标准分支的代码事实
- 不把它当成 `main` 分支正式稳定承诺
- 后续 `1.0` 收口时再决定是否覆盖 `main`

当前仍建议保守理解的尾项：

- `team / subAgent` 有过历史抽样和定向验证，但 live 黑盒还没到“全部正式收口”的程度
- 所以这条分支现在可以说“主链 local/bridge/hybrid 可测”，不能说“所有多代理场景已经完整验收”
